# Руководство по стилю и комментированию API

Этот документ задаёт единые правила оформления и комментирования для всех .proto файлов в репозитории.

## 1. Стандарты оформления (Google Protobuf Style)

Мы следуем официальному гайду Google для Protobuf. Источник: https://protobuf.dev/programming-guides/style/

Ключевые правила:

- Отступы: 2 пробела.
- Длина строки: 80 символов (предпочтительно).
- Имена:
  - Сообщения: CamelCase (например, `RegisterStartRequest`).
  - Поля: snake_case (например, `phone_number`).
  - Сервисы и RPC: CamelCase (например, `AuthService`, `LoginStart`).
  - Enum: UPPER_SNAKE_CASE (например, `AUTH_STATUS_OK`).
  - Значения enum: UPPER_SNAKE_CASE.
- Порядок:
  - `syntax`, затем `package`, затем `option`, затем `import`.
  - Объявления: service → message → enum (при необходимости).

Форматирование проверяется командой:

- make lint

## 2. Обязательное комментирование публичных элементов

Все публичные элементы должны иметь комментарии в стиле Protobuf (однострочные `//`).

Обязательные комментарии:

- service
- rpc методы
- message
- поля message
- enum
- значения enum

Требования к содержанию:

- Комментарий объясняет назначение и способ использования элемента.
- Для полей указываются:
  - формат и тип данных (например, E.164 для телефона, RFC3339 для дат),
  - ограничения (минимум/максимум, длина, допустимые значения),
  - поведение по умолчанию (если применимо).

Примеры:

- Для RPC: «Создаёт регистрацию и отправляет SMS. Возвращает `challenge_id` и срок действия кода».
- Для поля: «Телефон в формате E.164, обязательно, 7–15 цифр».

## 3. Правила версионирования

Мы используем SemVer на уровне сервиса. Версия отражается в имени пакета и пути:

- `package auth.v1;`
- путь: `proto/auth/v1/auth.proto`

Правила:

- Major: несовместимые изменения API.
- Minor: новые возможности без нарушения совместимости.
- Patch: исправления, не меняющие контракт.

При изменении версии:

- создаётся новая папка `vN` рядом с предыдущей;
- старые версии не модифицируются, только поддерживаются.

## 4. Соответствие текущим контрактам

Стиль и комментарии должны соответствовать уже существующим контрактам в:

- [proto/auth/v1/auth.proto](proto/auth/v1/auth.proto)

Если стиль расходится — корректируйте новые изменения, а не переписывайте историю без необходимости.
